# Manual de release: Salud ocupacional — Reportes (tag sugerido `v1.3.0`)

Este documento reúne la **historia** del desarrollo, el **alcance** de la versión y el **paso a paso** para fijar un tag en Git y publicarlo al remoto (mismo flujo que en releases anteriores del monorepo).

---

## 1. Historia (bitácora)

| Fase | Qué se hizo |
|------|----------------|
| **Base** | Pantalla **Reportes SO** (`/salud-ocupacional/reportes`): KPIs, condición al alta, top diagnósticos, atenciones por división, medicamentos, tendencia 12 meses, exportación PDF. API bajo `GET /api/salud-ocupacional/reports/*`. |
| **División** | Filtro por división alineado a textos SAP reales (no solo código `ARxx`): patrones + comparación **sin tildes** sobre división y subdivisión en snapshot. |
| **Catálogo UI** | En el selector de reportes se **excluyen AR40 y AR50** (`REPORTES_DIVISION_OPTIONS`). |
| **Top diagnósticos** | Se **eliminó la columna CIE-10** en UI y coherencia en PDF. |
| **Top pacientes** | Nuevo agregado: pacientes con más atenciones en el periodo; endpoint `top-patients`; corrección de agrupación solo por `patient_cod` y orden por `COUNT(*)`. |
| **Reincidencia** | KPI definido como pacientes con **≥ 2 consultas** en la misma ventana de fechas, sobre pacientes distintos. Texto de UI y PDF **simplificado** para usuario final (sin fracción técnica en el cuerpo principal). |
| **Subdivisión** | Segundo filtro opcional; lista vía `GET .../reports/subdivisions`; misma ventana de fechas + división opcional; SAP `txt_subdiv` para conteo de activos cuando aplica. |
| **RBAC** | Ver reportes: app **`reportes-so`**, feature **`ver`**, acción **`read`**. Perfiles seed **Enfermera** y **Admin SO (`jefe-so`)** incluyen acceso; ver sección 4. |

## 3. Paso a paso: commit, push y tag

Trabajá en la rama acordada (ej. `main`). Reemplazá `v1.3.0` por el número que definan (el último tag conocido en repo fue `v1.2.1`; **`v1.3.0`** sugiere versión menor nueva por bloque de funcionalidad).

### 3.1 Revisar cambios

```powershell
cd c:\Proyecto\Cursor\sami-V2
git status
git diff --stat
```

Confirmá que no queden archivos basura (por ejemplo `apps/backend/dist/` compilado local); **no** subir `dist/` al commit si el proyecto lo ignora con `.gitignore`.

### 3.2 Comprobar build (recomendado)

```powershell
pnpm build
```

### 3.3 Añadir y commitear

```powershell
git add -A
git commit -m "feat(so): reportes salud ocupacional, filtros división/subdivisión y PDF"
```

(Ajustá el mensaje al estilo del equipo; convención Conventional Commits opcional.)

### 3.4 Subir la rama

```powershell
git push origin main
```

(Si la rama remota tiene otro nombre, usala en lugar de `main`.)

### 3.5 Crear el tag anotado

Tag **ligero** (solo nombre):

```powershell
git tag v1.3.0
```

Tag **anotado** (recomendado: incluye autor, fecha y mensaje para `git show v1.3.0`):

```powershell
git tag -a v1.3.0 -m "Salud ocupacional: reportes, filtros división/subdivisión, PDF y KPIs"
```

### 3.6 Subir el tag al remoto

```powershell
git push origin v1.3.0
```

Para subir **todos** los tags locales pendientes:

```powershell
git push origin --tags
```

### 3.7 Verificación

```powershell
git tag -l "v1.3*"
git show v1.3.0 --no-patch
```

En GitHub/GitLab: comprobar que el tag aparece en **Releases** o en la lista de tags del repositorio.

## 4. Quién puede ver Reportes (RBAC)

- Ruta protegida en frontend: `assertAppAccess(session, 'reportes-so')`.
- Backend: `@RequirePermission('reportes-so', 'ver', 'read')` en los endpoints de reportes.
- Seed: roles `reportes-so:enfermera` y `reportes-so:jefe-so` con permiso sobre `reportes-so:ver`.
- Perfil **Admin SO** (`jefe-so`) incluye `reportes-so:jefe-so` y gestión del módulo.

Comprobación rápida: `GET /api/auth/me` → `appRoles` con `reportes-so` o `managedModuleSlugs` con `salud-ocupacional`.

## 5. Despliegue en servidor (recordatorio)

No forma parte del tag en sí, pero tras `git pull` o checkout del tag en el servidor:

```bash
pnpm install --frozen-lockfile
pnpm --filter @sami/backend build
pnpm --filter @sami/backend db:migrate   # si hubo migraciones en ese commit
pnpm --filter @sami/frontend build
# copiar dist del frontend y reiniciar PM2 según docs/ops/deploy-produccion-solucionesti2.md
```

El rollback por tag **no revierte** migraciones de BD ya aplicadas.
